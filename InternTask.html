Screening Task — OpenAPI Spec Crawler
Build a system that discovers, indexes, versions, and maintains a live catalog of public OpenAPI specifications from across the web.
Context
At APIMatic, we transform raw API specifications into SDKs, docs, and developer portals. A core part of that pipeline is having reliable, up-to-date access to OpenAPI specs across the ecosystem. Your task simulates this real-world challenge: crawl sources that host OpenAPI specs, build a versioned catalog, and keep it fresh. This task evaluates engineering judgment, not just code. Think about edge cases, data integrity, and maintainability. Learn more about APIMatic →

Requirements
🔭
Source discovery
  • Use the GitHub Code Search API to find files named openapi.yaml, openapi.json, swagger.yaml, swagger.json, or {API NAME}.json
  • Search public repos using queries
  • Optionally seed with a configurable list of known GitHub repo URLs to bootstrap the crawl
  • For simplicity, you can limit the crawl to stop after ingesting a certain number of specs
🕷️
Spec parsing
  • Support OpenAPI 2.x (Swagger) and 3.x YAML/JSON
  • Extract: title, version, description, servers, paths count, tags
🌿
Version management
  • Track spec version changes over time
  • OPTIONAL: Use both info.version field and content hash to detect updates
  • OPTIONAL: Retain previous versions (immutable history)
  • OPTIONAL: Emit a changelog diff summary on update (added/removed paths)
🗄️
Catalog store
  • Persist the final catalog as a JSON file (catalog.json) and maintain a versioned store in your project's Github (no cloud DBs required)
  • File properties: id, source_url, title, latest_version, fetched_at, history[ ]
🔄
OPTIONAL: Update cycle
  • Configurable polling interval (default: 24 h)
  • Incremental re-crawl: only re-fetch if ETag / Last-Modified has changed
  • Retry with backoff on fetch failures; mark as stale after N retries
  • Summary report after each run (new, updated, unchanged, failed)
🛡️
Quality & ops
  • Structured logging (JSON preferred) with run IDs
  • Unit tests
  • A README.md with setup, architecture overview, and design decisions

Deliverables
📤 What to submit
✓ A GitHub repo (public or shared) with all source code in any Programming Language or Framework
✓ README.md covering setup, architecture, design decisions, and known tradeoffs
✓ A sample catalog.json output from an actual crawl run
✓ A short Loom/GIF demo (~2 min) showing: initial crawl → catalog output → simulated update → diff summary
✓ Passing test suite (make test or equivalent)
💡 Tip from the team
We care more about thoughtful tradeoffs than feature count. A crawler that handles 2 sources reliably, explains its versioning logic clearly, and has a well-structured codebase will score higher than one that covers 10 sources sloppily. Write code you'd want to maintain.
